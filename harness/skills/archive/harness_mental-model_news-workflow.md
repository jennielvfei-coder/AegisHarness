# Privacy‑Aware Personalization

## 执行逻辑
### When to Use
- 用户请求个性化内容（例如日报、学习进展简报），且工作流需要从外部数据源拉取数据。
- 用户档案中明确包含隐私、内容过滤或合规要求。
- 环境已知存在不可用的外部源（如国际新闻站点屏蔽、API 需匿名化调用）。

### Step-by-Step
1. **前置档案读取**  
   在发起任何外部 API 调用或网络请求前，先读取用户的隐私档案/个人规划文件（如 Obsidian 中的「个人档案/xxx.md」）。
2. **提取合规要求**  
   从档案中识别并提取：
   - 敏感话题黑名单
   - 允许的数据存储位置（仅本地、严禁上传公有云）
   - 外部 API 的使用限制（如禁止将用户个人身份信息传入请求参数）
3. **调整数据源编排**  
   根据合规要求过滤或替换外部源：
   - 若用户要求“无国际源”，则跳过 Reuters/BBC/AP，转用本地缓存或国内替代源（如财联社）。
   - API 调用仅使用与用户无关的资源标识（country、language、category），绝不传递姓名、邮箱等 PII。
   - 所有数据处理在本地完成，不将原始用户消息转发至外部服务。
4. **生成结果本地化存储**  
   将最终内容写入用户指定的私有知识库（如 `claude专属文件夹/news/`），确保不暴露于公开网络。
5. **异常时隐私兜底**  
   若外部源不可用，**不降级为可能泄露隐私的替代源**；直接跳过或使用本地历史摘要，避免因 failover 引入隐私风险。

### How to Verify
- 工作流日志中，档案读取操作发生在外网请求之前。
- 检查 MCP / WebFetch 调用的参数，确认不含用户个人身份字段。
- 输出文件路径确认为用户私有笔记库内的指定目录。

## 异常处理
### Edge Cases
- **档案不存在**：回退为无个性化通用简报，不尝试从用户历史中推断偏好（避免隐性隐私侵犯）。
- **外部源全部不可用**：仅输出预置的「今日无可靠外部数据，请稍后重试」回应，不做任何 fallback 外部尝试。
- **用户消息中包含敏感上下文**：禁止将消息内容作为 API 参数；仅使用预定义的静态查询词。

### Fallback
- 永远不回退到“将用户原句上传搜索”的方案。
- 优先使用本地缓存中的新闻摘要，若无缓存则生成“暂无内容”的消息。
