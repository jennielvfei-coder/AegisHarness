# 环境前置条件验证

## 执行逻辑

### When to Use
- 工作流包含多个串行/并行工具调用，且后续步骤依赖前置资源（API 连接、文件模板、权限配置）
- 任务描述中引用了需要特定 MCP 服务器或本地文件路径的操作
- 上一次类似任务因环境问题失败，需要进行根因排查后的重试

### Step-by-Step

1. **识别依赖清单** — 解析工作流涉及的资源类型：
   - MCP 服务器（如 `world-news-api`）→ 需验证连接状态
   - 本地模板/配置文件 → 需验证路径存在且可读
   - 权限敏感操作（Write/WebFetch）→ 需确认 `settings.local.json` 已预授权
   - 特定模型能力（如 WebSearch）→ 需确认当前模型是否支持

2. **并行验证** — 对独立依赖同时检查：
   - MCP：`claude mcp list` 检查目标服务器状态为 `✓ Connected`
   - 文件：`Read` 关键文件头部确认存在性和版本号
   - 权限：读取 `settings.local.json` 中相关配置项
   - 能力：若已知模型不支持某功能（如 deepseek 的 WebSearch），标记为"已知失败模式"并准备降级路径

3. **生成状态报告** — 结构化输出：
   - ✅ 就绪项（直接使用）
   - ⚠️ 降级可用项（功能受限但可继续）
   - ❌ 阻塞项（必须修复才能继续）
   - 对每个 ❌ 项给出具体修复指令

4. **条件放行** — 仅当无 ❌ 阻塞项时进入工作流主体；否则向用户报告阻塞原因并等待修复

### How to Verify
- 工作流主体执行期间未出现因环境缺失导致的工具调用错误
- 所有 MCP 调用在首次尝试即成功返回
- 无权限弹窗中断自动化流程

## 异常处理

### Edge Cases
- **MCP 已列出但状态为 Disconnected**：尝试重新连接或提示用户手动重启 MCP 服务器
- **文件存在但版本不匹配**：若模板有新版本，读取新版本并提醒差异
- **权限配置声明存在但实际未生效**：重新读取配置文件验证键名拼写和路径格式

### Fallback
- 若 MCP 验证失败且无法自动恢复 → 提示用户使用备选数据源（如 WebFetch 替代 API）
- 若权限未预授权 → 在工作流开始前一次性列出所有需要的权限请求，减少中断次数
- 若整体环境不可用 → 记录失败模式到 memory，供下次 SessionStart 自动注入为"已知失败模式"
