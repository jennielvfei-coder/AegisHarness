# 验证优先于架构 (Verify Before Architect)

## 执行逻辑

### When to Use
当用户对现有系统提出一组诊断问题，并要求给出架构方案或改进建议时，**必须先验证、再架构**。此模式适用于：
- 代码库复杂且可能已发生变化（距上次分析超过数天）
- 诊断假设基于日志或行为观察，而非直接代码审查
- 多个问题之间存在相互依赖，需要确认连接关系

**触发信号：** 用户说"我诊断了五个问题……"或"按照铁律一——先验证，再架构……"

### Step-by-Step

#### Phase 1: 假设代码化（30秒）
将用户的每个诊断问题转化为可验证的代码断言：
```
问题1: "预言编译器缺失"
  → 验证断言: grep(pattern="prophet|Prophet|prediction|forecast", path=项目根)
  → 成功标准: 找到或未找到 Prophet 导入/调用
```

#### Phase 2: 并行验证（批量工具调用）
对每个断言，**一次性发出所有只读查询工具**：
- `Grep`：搜索关键符号是否存在
- `Read`：读取核心模块的入口文件（`__init__.py`、主模块文件）
- `codegraph_context`：追踪模块间的实际调用链（maxNodes=15~20）
- 避免逐个串行查询——批量发出以节省往返

**关键原则：** 此阶段只读、只观察、不修改。

#### Phase 3: 差距分析
将验证结果与用户诊断对照：
- ✅ 确认：代码现状与诊断一致
- ❌ 否认：代码已有该功能，诊断过时
- ⚠️ 部分：功能存在但实现方式不同于预期

**输出格式：**
| 问题 | 诊断 | 代码验证 | 状态 |
|------|------|----------|------|
| 1 | 预言编译器缺失 | `grep prophet` 无结果 | ✅ 确认 |
| 2 | 中文NER依赖字典 | `preprocess.py` 存在 jieba 分词 | ✅ 确认 |

#### Phase 4: 方案架构（仅在验证完成后）
基于验证结果给出：
1. **保持不变的模块**（验证确认功能正常的部分）
2. **需要修改的模块**（按依赖关系排序）
3. **新增模块**（接口定义清晰）
4. **修改影响分析**（哪些下游模块会受影响）

### How to Verify
- Phase 2 中每个断言都有明确的代码搜索结果
- Phase 3 的对照表覆盖所有用户提出的诊断点
- 如果某个诊断无法验证（例如涉及运行时行为），标记为"待运行时确认"而非跳过

## 异常处理

### Edge Cases
- **代码库不在当前工作区：** 询问路径或改为基于文档验证
- **诊断假设涉及运行时行为：** 标记为"需运行时验证"，先验证可静态检查的部分
- **验证结果与诊断全部矛盾：** 暂停架构，回传验证结果让用户重新评估诊断
- **代码量过大无法全部读取：** 优先读取 `__init__.py`、配置文件、核心逻辑入口，用 Grep 覆盖其余

### Fallback
如果工具调用全部失败（网络中断、权限不足），降级为：
1. 明确告知无法验证
2. 基于用户提供的诊断直接给出架构建议
3. 在方案中标注所有"未经代码验证"的假设
