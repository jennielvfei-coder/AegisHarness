# 并行信息收集（四工具调试）

## 执行逻辑
### When to Use
- 开始调查一个问题，尤其是面对不熟悉的代码库时
- 需要同时从多个角度获取信息，避免反复切换工具
- 时间敏感，希望一次调用拿到多维度背景

### Step-by-Step
1. **Read** – 打开一个关键配置文件、入口脚本或最近修改的文件，直接获取内容
2. **Grep** – 搜索与问题相关的符号、函数名或模式，定位代码位置和结构
3. **Bash** – 运行一个诊断命令（如包版本、服务状态、简单测试），验证环境假设
4. **MCP 工具** – 调用外部搜索或代码图谱工具（如 Tavily/CodeGraph），补充全局上下文

以上四个步骤**并行发起**，无需等待前一个完成。全部返回后拼接分析结果，快速形成对问题的整体认知。

### How to Verify
- 检查每个工具返回的状态：至少 3 个成功，失败项已记录
- 找到至少一条直接关联问题的线索（错误日志、异常函数、缺失配置等）
- 所有收集到的信息足够支撑下一步具体修复或深入排查

## 异常处理
### Edge Cases
- **工具部分失败**（如文件不存在、MCP 超时）：记录失败信息，不影响其他并行调用；后续再单独处理失败项
- **Grep 无匹配**：可能是搜索模式错误，立即调整模式重新搜索，或改用更宽泛的符号
- **Bash 命令需要交互**：优先选用非交互命令；若必须交互，改用预期输入重定向

### Fallback
- 若 MCP 工具不可用，使用内置 `web_search` 或 `codebase_search` 替代
- 若 Bash 权限不足，提示用户手动执行并反馈结果
- 若所有并行调用均失败，执行逐步单串回退：Read → Grep → Bash → MCP，保持顺序
